# გუნდის Deep Interview — სტრატეგია

## რატომ ვაკეთებთ (არა "გუნდი კმაყოფილია?")

**მიზანი:** Level 4 system-ისთვის data შეგროვება — ვინ რა როლში ჯდება AI-augmented ბივიჟენში. არა HR survey, არამედ capacity + knowledge mapping.

**ჰიპოთეზები (ტესტირებადი):**
1. დათოს ტექნიკური monopoly = #1 bottleneck (CLAUDE.md-ში L2-3). თუ დათო unavailable → development stops.
2. ინგას support system მუშაობს (L3), მაგრამ ინგაზე dependent. Knowledge transfer = 0.
3. მარი მაღ = hidden champion data (account manager + client relationships). Churn prevention-ის გასაღები.
4. მარი მიკ = development capacity რომელიც underused ან მისამართი არასწორი.
5. ლუკა = fresh eyes, AI-native generation. ყველაზე ადვილად შეითვისებს AI tools.

**შედეგი:** "გელა, აი რა გამოვიდა: [insight] → [action] → [system change]"

---

## ფაზა 1: ინგა + დათო (პარტნიორები)

**რატომ პირველი:** პარტნიორები არიან, სტრატეგიული view აქვთ, პროცესი გაიგებენ ზედმეტი ახსნის გარეშე. მათი cooperation = დანარჩენის legitimacy.

### ინგა ნადარია — Support / Operations

**ჰიპოთეზა:** ინგას სისტემა მუშაობს მაგრამ ინგაზეა dependent. Knowledge transfer = 0. + ინგას აქვს client health signals რომლებიც გელასთან არ მიდის.

**თემები:**
1. **Client pulse** — რომელ კლიენტთან ხშირია პრობლემა? ვისთან არ არის კონტაქტი თვეებია? (= churn early warning)
2. **Support patterns** — ერთი და იგივე პრობლემა მეორდება? რა ავტომატიზდება? (= system vs manual)
3. **Knowledge transfer** — თუ ინგა 2 კვირა არ არის, რა ხდება? ვინ ანაცვლებს? (= L3→L4 gap)
4. **გუნდის dynamics** — ვინ ვისთან მუშაობს ყოველდღე? სად არის friction?
5. **AI რა დაეხმარებოდა** — ticket routing? auto-response? client health alerts?

### დათო (დავით გაბისონია) — Development / Technical

**ჰიპოთეზა:** ტექნიკური monopoly. ყველა product-ის knowledge დათოს თავშია. + Bank API (Mardaleishvili) = delayed because overloaded.

**თემები:**
1. **Capacity** — რამდენ პროექტზე მუშაობ ერთდროულად? რა priority? (= bottleneck mapping)
2. **Technical debt** — რა "მუშაობს მაგრამ ნებისმიერ მომენტში გატყდება"? (= risk)
3. **Knowledge sharing** — მარი მიკ და ლუკა რამდენად autonomous არიან? რა სჭირდებათ? (= L2→L3 path)
4. **ERP connectors** — Fina/Oris/Balance/1C — maintenance effort? ახალი connector რამდენი ძალისხმევაა? (= moat strength)
5. **AI/automation** — Claude API + Qlik integration — feasible? რა prototype? (= AI positioning capability)
6. **Bank API (Mardaleishvili)** — status, blocker, timeline (= churn intervention)

## ფაზა 2: მარი მიკ + მარი მაღ + ლუკა

მას შემდეგ რაც ინგა/დათოსგან სტრუქტურა გვექნება.

### მარი მაღალკელიძე — Account Manager + Dev

**ჰიპოთეზა:** კლიენტების 70%-ს პირადად იცნობს. Champion redundancy-ის გასაღები.

**თემები:**
1. **Client relationships** — ვის ელაპარაკები რეგულარულად? ვისთან არ არის კონტაქტი? (= churn signals)
2. **Account health** — ვინ active-ად იყენებს? ვინ "იხდის მაგრამ არ ხსნის"? (= usage data)
3. **Dev role** — development-ში რას აკეთებ? overlap დათოსთან? (= capacity)
4. **AI tools** — რა დაგეხმარებოდა client management-ში? (= AI system design)

### მარი მიკურტუმაშვილი — Developer

**თემები:** capacity, skills, autonomy level, რა სჭირდება დათოსგან, AI tools readiness

### ლუკა — დამწყები

**თემები:** learning path, რას ხედავს fresh eyes-ით, AI comfort level, რა არ ესმის სისტემაში

## ფაზა 3: გელა (ოუნერი)

**რა ველოდები შენგან (დაფუშვა included):**

1. **Time allocation** — კვირაში რამდენ საათს ხარჯავ: გაყიდვებში, support-ში, development oversight-ში, admin-ში? (= L2 baseline)
2. **გადაწყვეტილებების bottleneck** — რა გადაწყვეტილებები ელოდება მხოლოდ შენ? რატომ ვერ იღებს სხვა? (= delegation gaps)
3. **ინგა/დათო როგორც პარტნიორები** — რეალურად პარტნიორული decision-making-ია თუ შენ წყვეტ ყველაფერს? (= L2 test)
4. **რას ვერ ანდობ არავის?** — ეს არის real bottleneck. "ყველაფერს ვანდობ" = არ ანდობ.
5. **AI C-Suite 1 თვის შემდეგ** — რა უნდა შეიცვალოს რომ თქვა "მუშაობს"? ციფრი, არა feeling.
6. **Balance.ge partnership** — email კამპანია = ლიდები. რამდენი? quality? follow-up ვინ აკეთებს?
7. **მარიამის კითხვარი** — შეავსო? გაუგზავნე ლინკი?
